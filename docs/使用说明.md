# TSched 使用说明

## 1. 项目简介

TSched 是一个面向 OpenWrt 和嵌入式 Linux 的轻量任务调度器，主要用于内部
开发、设备测试和自动化测试任务。项目使用 C99/gnu99 开发，核心运行时只依赖
libc 和 Linux 系统调用。

当前 V1 具备以下能力：

- 手动任务和固定间隔循环任务；
- 一项任务包含多条顺序 Shell 命令；
- 普通步骤失败后停止，`always_step` 清理步骤仍然执行；
- 最大运行轮次、执行超时和失败重试；
- 通过进程组取消任务及其派生进程；
- Unix Domain Socket 本机控制；
- `tschedctl` 命令行和 uHTTPd CGI 模式；
- 非阻塞 UDP 远程日志；
- 掉电后保留任务配置，运行历史和轮次不持久化；
- 启用的循环任务重启后重新进入调度。

V1 暂不支持 Cron、脚本上传、持久化运行历史、复杂 DAG 和可靠远程日志。

## 2. 平台要求

- Linux 内核 4.9 或更高版本；
- musl 或 glibc；
- GCC/Clang 和 CMake 3.10 以上版本用于本机构建；
- OpenWrt 构建时使用目标版本 SDK 或完整 Buildroot；
- Web 页面可选依赖 OpenWrt uHTTPd，核心守护进程不依赖 uHTTPd。

核心代码使用 `epoll`、`timerfd`、`signalfd`、Unix Domain Socket、`fork` 和
`waitpid`。未使用 pidfd、io_uring、cgroup v2 或其他 Linux 4.9 之后才提供的
关键接口。

## 3. 编译

### 3.1 快速构建

```sh
make
```

构建结果位于：

```text
build/tschedd
build/tschedctl
```

### 3.2 执行测试

```sh
make test
```

### 3.3 手动使用 CMake

```sh
cmake -S . -B build \
  -DCMAKE_BUILD_TYPE=MinSizeRel
cmake --build build
ctest --test-dir build --output-on-failure
```

发布到 OpenWrt 时应使用目标 SDK 交叉编译，不能直接复制本机 glibc 产物。

## 4. 目录和文件

建议部署布局：

```text
/usr/sbin/tschedd              守护进程
/usr/bin/tschedctl             CLI，同时可作为 CGI 程序
/etc/tsched/tsched.conf        全局配置
/etc/tsched/tasks.conf         持久化任务配置
/tmp/tsched.sock               默认控制 socket
/tmp/tsched/logs/              临时日志目录
/www/tsched/                   Web 静态文件
/www/cgi-bin/tsched-api        指向 tschedctl 的链接
```

`/etc/tsched` 在 OpenWrt 上通常位于可写 overlay，因此任务定义能够跨掉电保存。
`/tmp` 通常是 tmpfs，运行状态和临时日志不会消耗 Flash 擦写次数。

## 5. 启动守护进程

开发环境中以前台模式启动：

```sh
./build/tschedd ./etc/tsched.conf ./etc/tasks.conf
```

参数顺序为：

```text
tschedd [全局配置文件] [任务配置文件]
```

不传参数时默认读取：

```text
/etc/tsched/tsched.conf
/etc/tsched/tasks.conf
```

OpenWrt 包提供 procd init 脚本，安装后可执行：

```sh
/etc/init.d/tschedd enable
/etc/init.d/tschedd start
```

## 6. 全局配置

示例：

```ini
[global]
socket_path=/tmp/tsched.sock
task_file=/etc/tsched/tasks.conf
log_dir=/tmp/tsched/logs
max_running=16
startup_jitter_ms=5000

[udp_log]
udp_enabled=1
udp_host=192.168.1.100
udp_port=5514
```

字段说明：

| 字段 | 说明 | 默认值 |
|---|---|---|
| `socket_path` | CLI/CGI 与守护进程通信的 Unix Socket | `/tmp/tsched.sock` |
| `task_file` | 任务配置文件路径 | `/etc/tsched/tasks.conf` |
| `log_dir` | 临时日志目录；当前 V1 仅创建目录 | `/tmp/tsched/logs` |
| `max_running` | 最大同时运行任务数；`0` 表示不设置用户上限 | `16` |
| `startup_jitter_ms` | 重启后循环任务的确定性启动抖动 | `5000` |
| `udp_enabled` | 是否发送 UDP 日志 | `0` |
| `udp_host` | UDP 日志服务器 IP 或主机名 | 空 |
| `udp_port` | UDP 日志服务器端口 | `0` |

建议在资源有限设备上保留明确的 `max_running`，避免错误配置同时创建过多
子进程。填写数字 IP 可避免启动过程中依赖 DNS。

## 7. 任务配置

任务通过 `[task:ID]` section 定义：

```ini
format_version=1

[task:1]
name=network-test
enabled=1
schedule=interval
interval_ms=60000
max_runs=0
timeout_ms=30000
retry_count=1
workdir=/root/tests
step=./setup.sh
step=./run-test.sh
always_step=./cleanup.sh
```

字段说明：

| 字段 | 说明 |
|---|---|
| `name` | 任务名称，最长 63 字节 |
| `enabled` | `1` 启用，`0` 禁用 |
| `schedule` | `manual` 或 `interval` |
| `interval_ms` | 循环任务两轮之间的等待时间，单位毫秒 |
| `max_runs` | 本次守护进程生命周期内最多执行轮次，`0` 表示无限 |
| `timeout_ms` | 单轮任务整体超时，`0` 表示不限制 |
| `retry_count` | 失败后最多重试次数 |
| `workdir` | 执行 Shell 前切换到的工作目录 |
| `step` | 普通步骤；前一步失败后不再执行后续普通步骤 |
| `always_step` | 清理步骤；使用 `;` 接续，适合 teardown |

一项任务最多包含 16 个步骤，单条命令最长 1023 字节。V1 将步骤组合后交给
`/bin/sh -c` 执行，因此可以使用 BusyBox `ash` 支持的管道、重定向和变量。

### 7.1 循环与重启语义

循环任务使用 fixed-delay 语义：

1. 执行当前一轮；
2. 当前轮结束；
3. 等待 `interval_ms`；
4. 执行下一轮。

设备掉电或守护进程重启后：

- 任务配置和 `enabled` 状态仍然保留；
- 当前 PID、待运行实例、重试进度和运行历史丢失；
- `run_count` 从 0 开始；
- 启用的循环任务等待一个完整间隔后恢复；
- 首次等待时间还会加上 `task_id % startup_jitter_ms`；
- 不补偿掉电期间错过的轮次。

因此 `max_runs=100` 表示每次守护进程启动后最多执行 100 轮，而不是设备整个
生命周期累计 100 轮。

## 8. 命令行操作

默认连接 `/tmp/tsched.sock`：

```sh
./build/tschedctl status
./build/tschedctl list
./build/tschedctl run 1
./build/tschedctl cancel 1
./build/tschedctl stop
```

使用自定义 socket：

```sh
./build/tschedctl -s /tmp/test-tsched.sock list
```

命令说明：

| 命令 | 作用 |
|---|---|
| `status` | 显示任务数、运行数和调度堆任务数 |
| `list` | 列出任务 ID、名称、启用状态、运行状态和当前轮次 |
| `run ID` | 立即运行指定任务，包括 `manual` 任务 |
| `cancel ID` | 向任务进程组发送 `SIGTERM` |
| `stop` | 请求守护进程正常退出 |

## 9. UDP 远程日志

启用后，守护进程将任务生命周期和输出发送到一个共享 UDP 目标：

```text
TSCHED v=1 task=1 name="network-test" event=start ts=... msg=""
TSCHED v=1 task=1 name="network-test" event=output ts=... msg="..."
TSCHED v=1 task=1 name="network-test" event=finish ts=... msg="status=success code=0"
```

UDP 日志具有以下语义：

- 单个报文最多约 1200 字节；
- 使用非阻塞 `sendto`；
- 不确认、不重传、不写入 Flash；
- 日志服务器不可达不会改变任务结果；
- 报文可能丢失、重复或乱序；
- V1 长输出超过单个报文时会被截断。

可使用以下方式临时接收日志：

```sh
nc -u -l -p 5514
```

不同 `nc` 实现的参数可能不同，请以目标日志服务器为准。

## 10. Web 页面

Web 静态文件位于 `/www/tsched`，CGI 路径为：

```text
/www/cgi-bin/tsched-api -> /usr/bin/tschedctl
```

访问：

```text
http://设备地址/tsched/
```

CGI 检测到 `GATEWAY_INTERFACE` 后会切换到 JSON 输出模式。使用非默认 socket
时，应在 CGI 环境中设置：

```text
TSCHED_SOCKET=/path/to/tsched.sock
```

当前页面支持查看任务和手动触发。任务编辑、配置保存和远程日志参数编辑仍属于
后续版本功能。

## 11. 故障排查

### 无法连接 socket

```text
/tmp/tsched.sock: No such file or directory
```

检查守护进程是否启动、配置文件是否正确以及 socket 路径是否一致：

```sh
ps | grep tschedd
./build/tschedctl -s /实际路径 status
```

### 任务立即失败

- 检查 `workdir` 是否存在；
- 检查命令是否兼容目标设备上的 BusyBox `ash`；
- 检查命令依赖的程序是否安装；
- 查看守护进程标准输出和 UDP 日志；
- 退出码 `126` 通常表示切换工作目录或执行准备失败；
- 退出码 `127` 通常表示 Shell 或命令不存在。

### 循环任务重启后没有立即运行

这是默认设计。循环任务会等待 `interval_ms` 和启动抖动后再执行，以避免设备
启动时大量任务同时运行。

## 12. 当前限制

- 控制协议一次连接只处理一条短命令；
- 任务表最多 256 项；
- 单任务最多 16 个步骤；
- 运行历史仅存在于当前进程内存；
- 尚未实现配置 CRC、`.bak` 自动恢复和目录 `fsync`；
- 尚未实现本地有界日志文件；
- 超时当前直接发送 `SIGKILL`，尚未加入 `SIGTERM` 宽限期；
- Web 页面尚不能创建、编辑或删除任务；
- UDP 日志是尽力传输，不适合必须完整留存的审计场景。

这些限制会在后续里程碑逐步处理，设计细节参见
[设计文档](./设计文档.md)。
