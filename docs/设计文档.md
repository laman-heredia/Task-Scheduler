# TSched 设计文档

## 1. 设计目标

TSched 面向剩余 Flash 约 4 MB、内存约 128 MB 的 OpenWrt 设备。主要使用场景
是内部开发和自动化测试，而非公网多租户服务。

设计优先级从高到低为：

1. 低依赖和较小安装体积；
2. 空闲时低 CPU 占用；
3. 明确且可预测的内存、进程和文件描述符上限；
4. Linux 4.9、musl 和旧 OpenWrt 工具链兼容性；
5. 掉电后任务配置可恢复；
6. 自动化测试所需的命令、超时、重试和清理能力；
7. Web 操作便利性。

不将分布式调度、高可用、强一致运行历史、复杂权限系统和通用工作流引擎纳入
V1 范围。

## 2. 总体架构

```text
                  +----------------------+
Browser ----------> uHTTPd static + CGI |
                  +-----------+----------+
                              |
tschedctl CLI ----------------+  Unix Domain Socket
                              |
                    +---------v----------+
                    |      tschedd       |
                    |                    |
                    | epoll event loop   |
                    | scheduler min-heap |
                    | process manager    |
                    | UDP log sender     |
                    +----+-----------+---+
                         |           |
                 child process    UDP server
```

`tschedd` 是唯一常驻组件。CLI 和 CGI 都是短生命周期客户端，不持有任务状态。
核心守护进程不依赖 uHTTPd，因此没有 Web 需求的设备可以只安装核心包。

## 3. 组件职责

### 3.1 `src/daemon.c`

负责：

- 创建 `epoll`、`timerfd` 和 `signalfd`；
- 维护任务调度最小堆；
- 接受 Unix Socket 控制请求；
- `fork` 和回收任务子进程；
- 读取任务 stdout/stderr；
- 检查超时、处理重试并安排下一轮；
- 将任务事件转发到 UDP 日志服务。

当前实现将这些职责集中在单文件中以降低初始抽象成本。后续功能增加时，应拆分
为 scheduler、executor、ipc_server 和 event_loop 模块，同时保持单线程状态
所有权。

### 3.2 `src/config.c`

负责解析全局配置和任务配置，并提供原子保存接口。解析器是受限 INI 解析器：

- 单遍逐行读取；
- 固定行缓冲；
- 忽略未知 section 和未知字段；
- 不支持递归 include；
- 不引入第三方配置或 JSON 库。

### 3.3 `src/heap.c`

实现固定容量最小堆，排序键为任务的 `next_run_ms`：

- 查看最近任务：`O(1)`；
- 插入任务：`O(log N)`；
- 弹出任务：`O(log N)`；
- 最大容量与任务表相同，为 256。

当前任务调度模式只会在任务结束后重新插入，不需要任意位置删除。后续支持在线
修改调度时，可为任务增加 heap index 以支持 `O(log N)` 更新和删除。

### 3.4 `src/client.c`

同一二进制提供两种入口：

- 普通进程环境：作为 `tschedctl` CLI；
- 存在 `GATEWAY_INTERFACE`：作为 uHTTPd CGI。

两种入口复用 Unix Socket 请求代码，减少 Flash 占用和协议实现重复。

### 3.5 `src/log.c`

创建一个共享的非阻塞 UDP Socket，并发送文本格式事件。UDP 远程日志是旁路
功能，必须满足：

- 日志服务器故障不能阻止任务运行；
- 不进行无限内存缓存；
- 不因重试写入 Flash；
- 不把发送失败视为任务失败。

## 4. 事件循环

守护进程采用单线程 Reactor：

```text
epoll_wait
  |
  +-- timerfd   -> 弹出到期任务并启动
  +-- signalfd  -> 回收子进程或关闭服务
  +-- server fd -> accept 控制连接
  +-- client fd -> 解析一条控制命令并关闭
  +-- output fd -> 转发任务输出
```

单线程的优势：

- 不需要锁和原子操作；
- 不需要为线程预留额外栈空间；
- 状态变更顺序清晰；
- 更容易控制资源上限。

事件标识存储在 `epoll_event.data.u64`：

- 小整数表示 timer、signal 和 server；
- 高 32 位表示事件类型；
- 低 32 位保存任务 ID 或客户端连接槽位。

## 5. 时间模型与调度语义

### 5.1 单调时钟

固定间隔和超时使用 `CLOCK_MONOTONIC`，原因是：

- NTP 校时不会改变已经安排的等待时长；
- 用户修改系统墙钟不会导致任务提前或延迟；
- 适合“本轮结束后等待 N 毫秒”的语义。

### 5.2 最小堆与 timerfd

调度器使用一个共享 `timerfd`，每次设置为以下时间点中的最早者：

- 最小堆堆顶任务的 `next_run_ms`；
- 运行任务的 `started_ms + timeout_ms`；
- 已发送 `SIGTERM` 任务的强制终止宽限期。

定时器触发后：

1. 读取并清空 timerfd；
2. 弹出所有 `next_run_ms <= now` 的任务；
3. 启动可运行任务；
4. 并发名额不足时按触发顺序加入固定容量 FIFO；
5. 按新堆顶重新设置 timerfd。

与每秒扫描全部任务相比，该方案在空闲时不产生无意义遍历。
主事件循环可以无限期阻塞在 `epoll_wait`，不再依赖 1 秒轮询检查超时，因此
毫秒级间隔和任务超时不会额外引入一个固定轮询周期的抖动。

待运行队列只保存任务 ID，不保存任务数组指针，因此在线删除或移动任务配置不会
留下悬空指针。队列容量与最大任务数相同（256）；任务结束时直接派发队首任务，
避免固定周期重试造成额外唤醒和最长一个轮询周期的启动延迟。

### 5.3 fixed-delay

循环任务在当前轮执行结束后才计算：

```text
next_run = monotonic_now + interval_ms
```

这可以避免任务执行时间大于间隔时积累大量待运行实例，适合设备测试任务。

### 5.4 重启恢复

持久化任务配置不保存运行时状态。启动时：

```text
next_run = monotonic_now
         + interval_ms
         + task_id % startup_jitter_ms
```

因此：

- 无限循环任务可以在重启后继续循环；
- 有限轮次从 0 重新计数；
- 不追赶掉电期间错过的轮次；
- 多个任务不会必然在服务启动瞬间同时运行。

运行轮次属于一次测试周期的内存状态。守护进程重启会开始新周期；任务发生
`enabled: 0 -> 1` 转换时也会重置 `run_count` 和重试进度，允许开发人员显式
重新测试。对已经启用的任务重复执行 enable 不会绕过 `max_runs`。

## 6. 任务执行模型

### 6.1 逐步骤状态机

任务不再拼接为一段 Shell 字符串，而是由守护进程逐条启动：

```text
TASK_START
  -> NORMAL_STEP
  -> NORMAL_STEP
  -> ALWAYS_STEP
  -> ALWAYS_STEP
  -> TASK_FINISH
```

- 每个步骤独立通过 `/bin/sh -c` 执行；
- 普通步骤失败后跳过剩余普通步骤；
- 所有位于后方的 `always_step` 仍按配置顺序执行；
- 保存本轮第一个失败退出码，后续清理成功不能覆盖原始失败；
- 如果此前没有失败，`always_step` 失败会使任务失败；
- 显式取消或任务级超时后不再启动新步骤；
- `timeout_ms` 从整轮任务开始时计算，不在步骤切换时重置。

步骤间不共享 Shell 变量、函数或 `cd` 结果；需要共享 Shell 上下文的命令应放在
同一个步骤或外部脚本中。当前步骤通过 `active_step` 和 `step_count` 暴露给 Web。

### 6.2 进程组

子进程启动后调用：

```c
setpgid(0, 0);
```

运行期分别保存直接子进程 `leader_pid` 和进程组 `process_group`。`leader_pid`
仅用于 `waitpid` 关联；即使 Shell leader 已退出，只要后代仍持有输出管道，
`process_group` 就继续保留。超时、取消和 shutdown 统一通过经过边界检查的函数
向 `-process_group` 发信号，严格禁止 `kill(0, ...)`。

守护进程启用 Linux child subreaper。Shell leader 退出后，不论输出管道是否已经
EOF，都会通过 PGID 检查残留后代；存在后代时立即向步骤进程组发送 `SIGTERM`，
等待宽限期后再发送 `SIGKILL`，并由守护进程回收被重新托管的孤儿后代。只有确认
进程组消失后才完成当前步骤，避免后台程序通过重定向 stdout/stderr 逃逸。

### 6.3 输出处理

stdout 和 stderr 当前合并到同一个非阻塞 pipe：

- pipe 读端注册到 epoll；
- 守护进程分块读取；
- 默认不复制到守护进程 stdout；启用后也只进行非阻塞、尽力复制；
- 同时尽力发送 UDP 日志；
- 不在内存中保存完整输出。

本地日志写入 `/tmp` 中按任务覆盖的有界文件，不将完整输出保存在堆内存。当前
每个任务保留最近一次日志，达到 `local_log_kb` 后停止写入；全部任务日志同时
受到 `local_log_total_kb` 全局配额限制，避免在 128 MB 设备上过度占用 tmpfs。
截断标记本身也受这两个配额约束。

### 6.4 超时

V1 超时处理流程为：

1. 达到 `timeout_ms` 后发送 `SIGTERM`；
2. 进入 `TERMINATING` 状态；
3. 等待 `kill_grace_ms`；
4. 仍未退出时发送 `SIGKILL`。

### 6.5 重试

任务失败且仍处于启用状态、守护进程未停止、终止原因不是用户取消或 shutdown，
并且 `retries_done < retry_count` 时，延迟 `retry_delay_ms` 后重新调度。重试成功或耗尽
后才增加任务轮次。重试状态和终止原因不写 Flash。

## 7. 状态模型

当前任务状态：

```text
DISABLED
WAITING
RUNNING
RETRY_WAIT
PENDING
```

典型转换：

```text
DISABLED --配置启用/启动加载--> WAITING
WAITING  --到期或手动运行-----> RUNNING
WAITING  --并发名额耗尽-------> PENDING
PENDING  --获得并发名额-------> RUNNING
PENDING  --取消或禁用---------> WAITING/DISABLED
RUNNING  --失败且允许重试-----> WAITING
RUNNING  --循环任务完成-------> WAITING
RUNNING  --手动任务完成-------> WAITING
```

失败任务进入退避期后会显式设置为 `RETRY_WAIT`；`PENDING` 表示已触发但正在等待
并发名额。两种状态均通过 CLI/CGI 列表和 Web 页面展示。

## 8. 配置持久化与掉电安全

任务配置和全局配置保存采用相同的流程：

```text
config.new
  -> 写入完整内容
  -> fflush
  -> fsync(file)
  -> 原文件改名为 config.bak
  -> rename(config.new, config)
  -> fsync(parent directory)
```

同一文件系统内 `rename` 具有原子替换语义，可避免读取半个配置文件。
`rename(config.new, config)` 是逻辑提交点；如果后续父目录同步失败，实现会尽力
恢复旧主文件并再次同步目录，避免业务层回滚内存后磁盘却保留新配置。

新文件包含 CRC32；启动时校验主文件，校验或解析失败时自动加载 `.bak`。任务
配置中的 `format_version` 如存在必须等于当前支持版本；旧版不含版本和 checksum
的配置仍可读取，以便平滑升级。后续还
可加入 generation，以便在多个可恢复文件同时存在时选择更新版本。

运行轮次、PID、重试进度和历史不写入持久 Flash，以降低写放大。

## 9. IPC 协议

当前 V1 使用短文本命令：

```text
PING
STATUS
LIST
GET <task-id>
UPSERT <tab-separated task fields>
ENABLE <task-id> <0|1>
DELETE <task-id>
RUN <task-id>
CANCEL <task-id>
LOG <task-id>
CONFIG
SETUDP <enabled> <host> <port>
STOP
```

响应：

```text
OK ...
ERR ...
```

设计约束：

- Unix Domain Socket 仅供本机 CLI/CGI 使用；
- 一条连接处理一条请求；
- 服务端按换行累积读取，不依赖一次 `recv()` 返回完整请求；
- 同时最多维护 16 个客户端连接，超过上限时立即返回 busy；
- 请求缓冲上限 32767 字节，用于携带最多 16 个任务步骤；
- LIST/CGI 响应缓冲上限 48 KiB，可容纳 256 个最大名称任务的列表；
- 单任务日志读取仍只返回最近约 4 KiB；
- 响应按连接动态分配，并通过 `EPOLLOUT` 增量发送，慢控制端不会同步阻塞调度；
- 不为客户端创建线程。

后续任务编辑会带来更大 payload，届时应升级为固定头部加长度字段的版本化协议，
避免依赖分隔符处理任意命令内容。

## 10. Web 与 CGI

浏览器只访问 uHTTPd：

```text
Browser -> uHTTPd -> tsched-api CGI -> Unix Socket -> tschedd
```

优点：

- 复用 OpenWrt 常见 Web 服务；
- 核心包不引入 HTTP/TLS 库；
- 无 Web 设备无需安装静态资源；
- CGI 崩溃不会破坏调度状态。

缺点是每次请求都会创建 CGI 进程，因此页面使用 5 秒低频轮询，不适合高频实时
日志。后续可以增加增量日志 API，但不建议在 V1 引入 WebSocket。

## 11. UDP 日志协议

文本格式：

```text
TSCHED v=1 task=<id> name="<name>" event=<event>
       ts=<seconds.milliseconds> msg="<message>"
```

事件类型：

- `start`：任务子进程已经创建；
- `output`：读取到 stdout/stderr 数据；
- `finish`：任务退出，消息中包含状态和退出码。

报文固定缓冲为 1200 字节。远程日志目标在启动时通过 `getaddrinfo` 解析，所有任务
复用一个 UDP socket。

当前协议已转义引号、反斜杠、换行、制表符和不可打印字节，并携带 run ID 和
step ID。stdout/stderr 仍未分流；后续协议版本可继续增加：

- host 或设备 ID；
- stream 类型；
- sequence；
- 分片编号；
- 固定容量 EAGAIN 发送队列和 drop 计数。

## 12. 资源模型

### 12.1 静态上限

| 资源 | 上限 |
|---|---:|
| 任务数量 | 256 |
| 单任务步骤 | 16 |
| 任务名称 | 63 字节 + `\0` |
| 单条命令 | 1023 字节 + `\0` |
| 工作目录 | 255 字节 + `\0` |
| 同时等待的 epoll 事件 | 每轮 32 |
| Unix Socket listen backlog | 16 |

任务数组按 8、16、32 等容量递增，步骤数组和命令字符串按实际配置动态分配。
空配置不再预留全部 256 个任务和 4096 个命令缓冲区；同时仍通过任务数、步骤数
和命令长度上限约束最坏内存占用。

### 12.2 Flash 策略

- 二进制动态链接 libc；
- 使用 `-Os`、function/data sections 和 linker GC；
- Web 页面不使用前端框架、图片、字体和 npm；
- 配置仅在用户修改时写 Flash；
- 运行状态和日志默认放在 `/tmp`；
- 不引入 SQLite、TLS、事件库和 JSON 第三方库。

## 13. 兼容性

为了兼容 Linux 4.9，禁止依赖：

- pidfd；
- io_uring；
- `clone3`；
- cgroup v2；
- time namespace；
- systemd API。

代码标准为 C99/gnu99，不使用 C11 原子、C11 threads 或 `_Generic`。Linux 专用
模块可以使用 `_GNU_SOURCE` 暴露 `accept4`、`SOCK_CLOEXEC` 等接口。

## 14. 主要风险与后续改进

建议后续按照以下优先级推进：

### V1 可靠性状态

V1 已完成配置 CRC/备份恢复、非阻塞 IPC 背压、客户端超时、输出字段转义和有界
资源控制。任务输出到 daemon stdout 也采用非阻塞尽力语义，二进制本地日志通过
文本转义后进入 CGI。配置 generation 和 UDP 长行分片不是当前掉电恢复或尽力日志语义的必要
条件，可在需要多代配置择新或远程日志重组时再作为兼容扩展加入。

### P1：可用性

1. 运行摘要环形缓冲；
2. CGI 按偏移增量读取日志。

### P2：调度能力

1. `once` 和 `delay`；
2. Cron；
3. fixed-rate 与错过任务策略；
4. 持久化轮次作为可选功能；
5. 上传脚本和测试资源。

## 15. 测试策略

当前自动化测试已覆盖最小堆、配置保存/损坏恢复、步骤状态机、进程组取消、任务
容量、毫秒调度、最大轮次、IPC 分片请求、慢客户端响应背压、UDP 转义以及 Web/CGI
多步骤任务。仍建议在通用 Linux CI 中继续增加：

- 更长时间、多并发的 stdout/stderr 持续输出压力；
- UDP `EAGAIN` 和高丢包计数；
- 32 位平台整数边界；
- OpenWrt musl 交叉编译；
- Linux 4.9 真机或虚拟机运行验证。
