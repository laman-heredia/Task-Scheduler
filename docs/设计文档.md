# TSched 设计文档

## 1. 设计目标

TSched 面向剩余 Flash 约 4 MB、内存约 128 MB 的 OpenWrt 设备。主要使用场景
是内部开发和自动化测试，而非公网多租户服务。

设计优先级从高到低为：

1. 低依赖和较小安装体积；
2. 空闲时低 CPU 占用；
3. 明确且可预测的内存、进程和文件描述符上限；
4. Linux 4.9、musl 和旧 OpenWrt 工具链兼容性；
5. 掉电后任务配置可恢复；
6. 自动化测试所需的命令、超时、重试和清理能力；
7. Web 操作便利性。

不将分布式调度、高可用、强一致运行历史、复杂权限系统和通用工作流引擎纳入
V1 范围。

## 2. 总体架构

```text
                  +----------------------+
Browser ----------> uHTTPd static + CGI |
                  +-----------+----------+
                              |
tschedctl CLI ----------------+  Unix Domain Socket
                              |
                    +---------v----------+
                    |      tschedd       |
                    |                    |
                    | epoll event loop   |
                    | scheduler min-heap |
                    | process manager    |
                    | UDP log sender     |
                    +----+-----------+---+
                         |           |
                 child process    UDP server
```

`tschedd` 是唯一常驻组件。CLI 和 CGI 都是短生命周期客户端，不持有任务状态。
核心守护进程不依赖 uHTTPd，因此没有 Web 需求的设备可以只安装核心包。

## 3. 组件职责

### 3.1 `src/daemon.c`

负责：

- 创建 `epoll`、`timerfd` 和 `signalfd`；
- 维护任务调度最小堆；
- 接受 Unix Socket 控制请求；
- `fork` 和回收任务子进程；
- 读取任务 stdout/stderr；
- 检查超时、处理重试并安排下一轮；
- 将任务事件转发到 UDP 日志服务。

当前实现将这些职责集中在单文件中以降低初始抽象成本。后续功能增加时，应拆分
为 scheduler、executor、ipc_server 和 event_loop 模块，同时保持单线程状态
所有权。

### 3.2 `src/config.c`

负责解析全局配置和任务配置，并提供原子保存接口。解析器是受限 INI 解析器：

- 单遍逐行读取；
- 固定行缓冲；
- 忽略未知 section 和未知字段；
- 不支持递归 include；
- 不引入第三方配置或 JSON 库。

### 3.3 `src/heap.c`

实现固定容量最小堆，排序键为任务的 `next_run_ms`：

- 查看最近任务：`O(1)`；
- 插入任务：`O(log N)`；
- 弹出任务：`O(log N)`；
- 最大容量与任务表相同，为 256。

当前任务调度模式只会在任务结束后重新插入，不需要任意位置删除。后续支持在线
修改调度时，可为任务增加 heap index 以支持 `O(log N)` 更新和删除。

### 3.4 `src/client.c`

同一二进制提供两种入口：

- 普通进程环境：作为 `tschedctl` CLI；
- 存在 `GATEWAY_INTERFACE`：作为 uHTTPd CGI。

两种入口复用 Unix Socket 请求代码，减少 Flash 占用和协议实现重复。

### 3.5 `src/log.c`

创建一个共享的非阻塞 UDP Socket，并发送文本格式事件。UDP 远程日志是旁路
功能，必须满足：

- 日志服务器故障不能阻止任务运行；
- 不进行无限内存缓存；
- 不因重试写入 Flash；
- 不把发送失败视为任务失败。

## 4. 事件循环

守护进程采用单线程 Reactor：

```text
epoll_wait
  |
  +-- timerfd   -> 弹出到期任务并启动
  +-- signalfd  -> 回收子进程或关闭服务
  +-- server fd -> accept 控制连接
  +-- client fd -> 解析一条控制命令并关闭
  +-- output fd -> 转发任务输出
```

单线程的优势：

- 不需要锁和原子操作；
- 不需要为线程预留额外栈空间；
- 状态变更顺序清晰；
- 更容易控制资源上限。

事件标识存储在 `epoll_event.data.u64`：

- 小整数表示 timer、signal 和 server；
- 高 32 位表示事件类型；
- 低 32 位保存任务 ID 或客户端 fd。

## 5. 时间模型与调度语义

### 5.1 单调时钟

固定间隔和超时使用 `CLOCK_MONOTONIC`，原因是：

- NTP 校时不会改变已经安排的等待时长；
- 用户修改系统墙钟不会导致任务提前或延迟；
- 适合“本轮结束后等待 N 毫秒”的语义。

### 5.2 最小堆与 timerfd

调度器只将堆顶任务的剩余时间设置到一个 `timerfd`。定时器触发后：

1. 读取并清空 timerfd；
2. 弹出所有 `next_run_ms <= now` 的任务；
3. 启动可运行任务；
4. 资源不足时延迟 500 ms 后重新入堆；
5. 按新堆顶重新设置 timerfd。

与每秒扫描全部任务相比，该方案在空闲时不产生无意义遍历。

### 5.3 fixed-delay

循环任务在当前轮执行结束后才计算：

```text
next_run = monotonic_now + interval_ms
```

这可以避免任务执行时间大于间隔时积累大量待运行实例，适合设备测试任务。

### 5.4 重启恢复

持久化任务配置不保存运行时状态。启动时：

```text
next_run = monotonic_now
         + interval_ms
         + task_id % startup_jitter_ms
```

因此：

- 无限循环任务可以在重启后继续循环；
- 有限轮次从 0 重新计数；
- 不追赶掉电期间错过的轮次；
- 多个任务不会必然在服务启动瞬间同时运行。

## 6. 任务执行模型

### 6.1 命令组合

V1 将任务步骤组成一段 Shell 脚本：

```text
step1 && step2 ; always_step
```

- `step` 使用 `&&`；
- `always_step` 使用 `;`；
- 最终通过 `/bin/sh -c` 执行。

这种方式代码量小，且适合内部测试脚本，但存在限制：多个 `always_step` 和复杂
失败传播难以表达。后续应改为逐步骤执行状态机，准确记录每个步骤结果。

### 6.2 进程组

子进程启动后调用：

```c
setpgid(0, 0);
```

任务 PID 同时作为进程组 ID。超时或取消时对 `-pid` 发信号，可以覆盖 Shell
启动的直接子进程和孙进程，减少测试进程残留。

### 6.3 输出处理

stdout 和 stderr 当前合并到同一个非阻塞 pipe：

- pipe 读端注册到 epoll；
- 守护进程分块读取；
- 输出到守护进程 stdout；
- 同时尽力发送 UDP 日志；
- 不在内存中保存完整输出。

后续实现本地日志时应写入 `/tmp` 中的有界文件，仍不得将完整输出保存在堆内存。

### 6.4 超时

V1 超时后直接发送 `SIGKILL`。目标方案应改为：

1. 达到 `timeout_ms` 后发送 `SIGTERM`；
2. 进入 `TERMINATING` 状态；
3. 等待 `kill_grace_ms`；
4. 仍未退出时发送 `SIGKILL`。

### 6.5 重试

任务失败且 `retries_done < retry_count` 时，延迟 1000 ms 后重新调度。重试成功或
耗尽后才增加任务轮次。重试状态不写 Flash。

## 7. 状态模型

当前任务状态：

```text
DISABLED
WAITING
RUNNING
RETRY_WAIT
```

典型转换：

```text
DISABLED --配置启用/启动加载--> WAITING
WAITING  --到期或手动运行-----> RUNNING
RUNNING  --失败且允许重试-----> WAITING
RUNNING  --循环任务完成-------> WAITING
RUNNING  --手动任务完成-------> WAITING
```

当前 `RETRY_WAIT` 枚举尚未被独立设置，重试仍复用 `WAITING`。后续拆分执行器时应
完善状态转换和 API 展示。

## 8. 配置持久化与掉电安全

任务配置保存的当前流程：

```text
tasks.conf.new
  -> 写入完整内容
  -> fflush
  -> fsync(file)
  -> rename(tasks.conf.new, tasks.conf)
```

同一文件系统内 `rename` 具有原子替换语义，可避免读取半个配置文件。

完整的掉电保护目标还应补充：

1. 原文件重命名为 `tasks.conf.bak`；
2. 新文件携带 generation 和 CRC32；
3. `rename` 后 `fsync` 父目录；
4. 启动时校验主文件；
5. 主文件损坏时加载 `.bak`；
6. 主文件和备份都损坏时保留现场并以空任务集启动。

运行轮次、PID、重试进度和历史不写入持久 Flash，以降低写放大。

## 9. IPC 协议

当前 V1 使用短文本命令：

```text
PING
STATUS
LIST
RUN <task-id>
CANCEL <task-id>
STOP
```

响应：

```text
OK ...
ERR ...
```

设计约束：

- Unix Domain Socket 仅供本机 CLI/CGI 使用；
- 一条连接处理一条请求；
- 请求缓冲上限 511 字节；
- 响应缓冲上限 4095 字节；
- 不为客户端创建线程。

后续任务编辑会带来更大 payload，届时应升级为固定头部加长度字段的版本化协议，
避免依赖分隔符处理任意命令内容。

## 10. Web 与 CGI

浏览器只访问 uHTTPd：

```text
Browser -> uHTTPd -> tsched-api CGI -> Unix Socket -> tschedd
```

优点：

- 复用 OpenWrt 常见 Web 服务；
- 核心包不引入 HTTP/TLS 库；
- 无 Web 设备无需安装静态资源；
- CGI 崩溃不会破坏调度状态。

缺点是每次请求都会创建 CGI 进程，因此页面使用 5 秒低频轮询，不适合高频实时
日志。后续可以增加增量日志 API，但不建议在 V1 引入 WebSocket。

## 11. UDP 日志协议

文本格式：

```text
TSCHED v=1 task=<id> name="<name>" event=<event>
       ts=<seconds.milliseconds> msg="<message>"
```

事件类型：

- `start`：任务子进程已经创建；
- `output`：读取到 stdout/stderr 数据；
- `finish`：任务退出，消息中包含状态和退出码。

报文固定缓冲为 1200 字节。远程日志目标在启动时通过 `getaddrinfo` 解析，所有任务
复用一个 UDP socket。

当前消息内容未完整转义，且 stdout/stderr 未分流；后续协议版本应增加：

- host、run ID、step ID；
- stream 类型；
- sequence；
- 分片编号；
- 对引号、反斜杠、换行和不可打印字符进行转义；
- 固定容量 EAGAIN 发送队列和 drop 计数。

## 12. 资源模型

### 12.1 静态上限

| 资源 | 上限 |
|---|---:|
| 任务数量 | 256 |
| 单任务步骤 | 16 |
| 任务名称 | 63 字节 + `\0` |
| 单条命令 | 1023 字节 + `\0` |
| 工作目录 | 255 字节 + `\0` |
| 同时等待的 epoll 事件 | 每轮 32 |
| Unix Socket listen backlog | 16 |

任务表直接位于 `daemon_state` 内，不会因配置任务数增加而频繁调用堆分配器。

### 12.2 Flash 策略

- 二进制动态链接 libc；
- 使用 `-Os`、function/data sections 和 linker GC；
- Web 页面不使用前端框架、图片、字体和 npm；
- 配置仅在用户修改时写 Flash；
- 运行状态和日志默认放在 `/tmp`；
- 不引入 SQLite、TLS、事件库和 JSON 第三方库。

## 13. 兼容性

为了兼容 Linux 4.9，禁止依赖：

- pidfd；
- io_uring；
- `clone3`；
- cgroup v2；
- time namespace；
- systemd API。

代码标准为 C99/gnu99，不使用 C11 原子、C11 threads 或 `_Generic`。Linux 专用
模块可以使用 `_GNU_SOURCE` 暴露 `accept4`、`SOCK_CLOEXEC` 等接口。

## 14. 主要风险与后续改进

建议后续按照以下优先级推进：

### P0：可靠性

1. 配置 CRC32、generation、`.bak` 和父目录 `fsync`；
2. `SIGTERM` + 宽限期 + `SIGKILL`；
3. 独立步骤状态机，正确保证全部 teardown 执行；
4. 有界待运行队列；
5. 客户端数量计数，而不是通过 fd 数值间接判断；
6. 输出内容转义和 UDP 长行分片。

### P1：可用性

1. 任务增删改查 IPC；
2. Web 任务编辑和 UDP 参数编辑；
3. `/tmp` 有界本地日志；
4. 运行摘要环形缓冲；
5. CGI 增量读取日志。

### P2：调度能力

1. `once` 和 `delay`；
2. Cron；
3. fixed-rate 与错过任务策略；
4. 持久化轮次作为可选功能；
5. 上传脚本和测试资源。

## 15. 测试策略

当前自动化测试覆盖最小堆顺序和基础配置解析。后续至少需要增加：

- 任务上限和步骤上限；
- 超长行、非法数字和重复任务 ID；
- 原子保存与模拟损坏恢复；
- 循环任务重启初始化；
- 最大轮次和失败重试；
- 超时终止整个进程组；
- 大量 stdout/stderr 的 pipe 背压；
- UDP 服务不可达和 `EAGAIN`；
- 32 位平台整数边界；
- OpenWrt musl 交叉编译；
- Linux 4.9 真机或虚拟机运行验证。
